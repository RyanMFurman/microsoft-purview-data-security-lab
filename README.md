# Microsoft Purview Healthcare Data Security & AI Governance Lab

> **Evidence boundary:** Free hybrid lab. No Microsoft 365 tenant was available or used. No Purview policy, label, alert, search, or investigation was deployed or observed in a live tenant.

## Evidence labels

- **IMPLEMENTED LOCALLY** — Code or workflows personally executed and validated on a local system.
- **DESIGNED FOR PURVIEW** — Technically detailed Microsoft Purview configuration that was not deployed.
- **SIMULATED INVESTIGATION** — Fictional evidence and events used to demonstrate an analyst workflow.

Every substantive artifact must use one of these labels. A design is never evidence of deployment.

## Project objective

Design and locally validate a defensible healthcare data-security program for fictional company Contoso Care Assist. The project demonstrates how an analyst would discover, classify, protect, investigate, retain, and report on sensitive Microsoft 365 data while preparing for Microsoft 365 Copilot.

## Operating model

**DESIGNED FOR PURVIEW** — Microsoft Purview configurations are specified using current Microsoft documentation and official SC-401 workflows.

**IMPLEMENTED LOCALLY** — Synthetic-data generation, sensitive-data analysis, configuration validation, test cases, diagrams, and reports run locally.

**SIMULATED INVESTIGATION** — Forensic events and findings are explicitly fictional and never presented as tenant telemetry.

## Business scenario

Contoso Care Assist is a fictional healthcare-technology company. It uses Exchange Online, Teams, SharePoint, and OneDrive conceptually to support patients, evaluate synthetic transcription output, collaborate internally, and manage employee and business records. Before a proposed Microsoft 365 Copilot pilot, it needs to reduce oversharing, establish classification and lifecycle controls, and define a repeatable investigation process.

All names, records, identifiers, organizations, and events in this repository are synthetic.
